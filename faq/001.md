## prompt

so given a physics paper and we want to test its claims, we can create new tests, plug in their equations and test against our physics engine test suite, is that right?




Yes — **that is the intended direction**, but with one important correction:

```text
We do not “test a paper” directly.
We test specific, formalized claims extracted from the paper.
```

A physics paper is prose + equations + assumptions + conventions. PTW/BMC should turn that into **claim fixtures** and **test obligations**.

## The correct pipeline

Think of the paper like a software library proposal.

The paper says:

```text
Given assumptions A, equations E imply behavior B in regime R.
```

PTW should convert that into:

```text
1. Claim extraction
2. Equation/convention mapping
3. Minimal executable fixture
4. Invariant/constraint tests
5. Null-model comparison
6. Numerical reliability checks
7. EBP debt report
```

So yes, for a given physics paper, we can create new tests that plug in its equations — but only after we define exactly what kind of claim we are testing.

## Example

Suppose a Bohmian cosmology paper claims:

```text
For this Wheeler-DeWitt wavefunction Ψ(α, φ), Bohmian trajectories recover classical FRW behavior at large scale factor.
```

PTW should not immediately say “test the paper.”

It should break that into smaller testable claims:

```text
Claim 1: Ψ satisfies the stated Wheeler-DeWitt equation.
Claim 2: The guidance equations are derived consistently from S = arg Ψ.
Claim 3: The trajectory remains finite in the claimed region.
Claim 4: The chosen clock variable is locally monotonic in that region.
Claim 5: Q / classical_term becomes small in the claimed limit.
Claim 6: The resulting trajectory approximates the stated classical FRW target.
Claim 7: The result survives step-size, threshold, and null-model tests.
```

Each claim becomes a fixture/test.

## What “plug in their equations” means

It means we create a **paper profile** like:

```text
paper_id: example_bohmian_cosmology_2024
claim_id: classical_limit_recovery_claim_01
equations:
  - wavefunction Ψ(α, φ)
  - Wheeler-DeWitt operator
  - guidance equations
  - classical target equation
assumptions:
  - units
  - factor ordering
  - minisuperspace metric
  - clock choice
  - boundary conditions
test_targets:
  - WdW residual
  - trajectory finiteness
  - clock readiness
  - Q/classical ratio
  - Friedmann residual
  - convergence
  - null comparison
```

Then the engine runs tests against that profile.

## But there are strict limits

Passing tests would mean:

```text
The claim survived these implemented checks under these assumptions.
```

It would **not** mean:

```text
The paper is proven true.
The theory is validated.
Bohmian cosmology is confirmed.
Quantum gravity is solved.
```

That distinction is the heart of EBP.

## The ideal PTW architecture

Eventually, yes, PTW should support something like:

```text
papers/
  bohmian_cosmology_example/
    claim_registry.json
    equations.go
    fixtures.go
    null_models.go
    tests/
      wdw_residual_test.go
      guidance_test.go
      convergence_test.go
      friedmann_residual_test.go
      null_comparison_test.go
```

But the paper’s equations should not be pasted blindly into tests. First they need:

```text
convention mapping
unit checks
domain assumptions
boundary conditions
parameter regimes
null models
known limiting cases
failure cases
```

Otherwise we risk creating a test that only confirms the paper’s own assumptions.

## Where BMC-POST-0001 and POST-0002 fit

These remediation tickets are building the minimum machinery needed for paper testing.

POST-0001 taught the system:

```text
Wrong WdW inputs should fail.
```

POST-0002 taught the report path:

```text
The numerical diagnostic, not just the analytic identity, should carry failure authority.
```

That is exactly the foundation we need before testing claims from real papers.
